SxCrypt Whitepaper

A Final-Stage Technical Whitepaper on Adaptive, Containment-Oriented Vault Defense

Bala

Final Version — 2026

Abstract

SxCrypt is a defensive cryptographic vault system built around modern, audited primitives, augmented by a reactor-inspired adaptive defense model. It does not claim to replace AES, nor does it claim that any custom block-cipher construction included with it is mathematically stronger than AES. Instead, SxCrypt asserts a precise and verifiable thesis:

AES protects ciphertext. SxCrypt builds a full defensive vault system around AES using adaptive reactor-inspired containment, attacker-cost escalation, decoy uncertainty, key zeroization, tamper evidence, and recovery-controlled lock states.

A failed access attempt does not damage external systems. Instead, it triggers an internal defensive chain reaction: computational cost grows, memory cost grows, delay grows, decoy uncertainty grows, and containment state becomes stricter. The system remains non-destructive, defensive-only, and vault-local at all times.

This whitepaper presents the formal mathematical model, threat model, architecture, state machine, key schedule, encryption flow, evaluation methodology, implementation rules, and safety boundaries of SxCrypt. It also presents SxCrypt-Core, a research-only AES-style SPN cipher used purely as a mathematical laboratory for studying substitution, permutation, diffusion, avalanche, and S-box quality.

The whitepaper closes with a brief personal dedication, kept structurally separate from all technical claims.

1. Introduction

1.1 Motivation

Modern symmetric encryption is, in isolation, extraordinarily strong. AES-256-GCM with a high-entropy key is not the weak link in any realistic deployment. The weak links are:

  1. Human passphrases of low entropy.
  2. Repeated guessing attempts against stolen vault blobs.
  3. Tampering with vault metadata or ciphertext.
  4. Replay of stale blobs.
  5. Lack of containment when an adversary obtains a copy of the vault.
  6. Side observation of failed attempts.
  7. Absence of an enforced cost-growth model against offline attackers.

SxCrypt addresses the system layer around the cipher, not the cipher itself.

1.2 Thesis

A failed attempt must increase internal defensive cost, never external damage.

This is the core law of SxCrypt. Every design decision in this document is derived from, or constrained by, that law.

1.3 Honest Positioning

SxCrypt is not:

  • A replacement for AES.
  • A proven improvement over AES as a block cipher.
  • A destructive countermeasure framework.
  • A network-active or self-propagating system.
  • A forensic eraser, wiper, or anti-forensic payload.

SxCrypt is:

  • A vault-local, non-destructive defensive system.
  • An adaptive reactor of computational and memory cost.
  • A formally specified state machine of containment levels.
  • A standards-based encryption pipeline (AES-256-GCM, Argon2id, HKDF-SHA512, HMAC-SHA512).
  • A research laboratory (SxCrypt-Core) for studying SPN cipher internals.

2. System Overview

SxCrypt is composed of three clearly separated layers:

+---------------------------------------------------------------+
|                  REACTOR DEFENSE MODEL                        |
|   adaptive cost, delay, decoys, containment, zeroization      |
+---------------------------------------------------------------+
|                       SxCrypt-Vault                           |
|   AES-256-GCM | Argon2id | HKDF-SHA512 | HMAC-SHA512          |
+---------------------------------------------------------------+
|                  SxCrypt-Core (RESEARCH ONLY)                 |
|   AES-style SPN: SubBytes, ShiftRows*, MixColumns*, ARK       |
+---------------------------------------------------------------+

2.1 SxCrypt-Core (Research Layer)

A research-only AES-style SPN block cipher used to study:

  • Byte-substitution via a GF(2^8) inverse plus affine transform.
  • ShiftRows-like permutation.
  • MixColumns-like GF(2^8) diffusion.
  • AddRoundKey and key expansion.
  • Avalanche, entropy, branch number, active S-box counts.

Supported parameters: 128 / 192 / 256-bit keys, 12 / 14 / 16 rounds.

Warning. SxCrypt-Core is experimental. It is not the default protection layer for real secrets. It is a mathematical laboratory, not a deployment cipher.

2.2 SxCrypt-Vault (Production Layer)

The production-oriented defensive vault, built exclusively from standard, audited primitives:

  • Argon2id for passphrase hardening.
  • AES-256-GCM for authenticated encryption.
  • HMAC-SHA512 for external integrity and tamper evidence.
  • HKDF-SHA512 for key separation.
  • Cryptographically secure random salts, nonces, vault IDs, and decoy IDs.

The vault must never:

  • Attack any external system.
  • Spread or self-replicate.
  • Delete external files.
  • Scan the host system.
  • Exfiltrate data.
  • Modify anything outside its own vault container.

2.3 Reactor Defense Model (Adaptive Layer)

The reactor is the adaptive control plane. Every wrong attempt raises attacker cost, waiting time, memory cost, decoy uncertainty, and containment level. The reaction is strictly internal:

  • It may zeroize temporary in-memory secrets.
  • It may lock the vault and require recovery.
  • It must not delete user files, system files, other vaults, folders, drives, or network data.

States used by the reactor (full definitions in §6):

OPEN, BACKOFF, QUARANTINE, SCRAM_LOCK, CONTAINMENT_LOCK, RECOVERY_REQUIRED, ZEROIZE_SESSION, DECOY_OPEN, TAMPER_EVIDENT.

Terminology. The word meltdown is intentionally avoided as a destructive system action. If used at all, it refers only to a visual metaphor for maximum internal containment — never deletion, damage, or any external effect.

3. Threat Model

3.1 Adversary Capabilities Considered

SxCrypt is designed against an attacker that may:

  • Obtain a copy of a sealed vault blob.
  • Run unbounded offline brute-force on it.
  • Submit many wrong passphrases interactively.
  • Automate guessing scripts.
  • Tamper with vault headers, nonces, associated data, or ciphertext.
  • Replay older vault blobs where policy makes replay relevant.
  • Inspect casual forensic metadata (filenames, timestamps).
  • Probe with timing-insensitive abuse patterns.

3.2 Adversary Capabilities Out of Scope

SxCrypt explicitly does not defend against:

  • A compromised endpoint while the vault is open.
  • Keyloggers capturing the passphrase.
  • Malware with root or administrator privileges.
  • Screen capture of unlocked plaintext.
  • Genuinely weak passphrases.
  • Social engineering.
  • Nation-state cold-boot or DRAM remanence attacks after unlock.
  • Side-channel attacks (power, EM, cache) unless separately hardened.
  • Malicious builds or backdoored source code.

3.3 Security Goals

  1. Confidentiality of sealed payloads under AES-256-GCM.
  2. Integrity and authenticity via GCM tag and an external HMAC-SHA512.
  3. Cost growth against repeated failed attempts.
  4. Containment that escalates monotonically with risk score.
  5. Tamper evidence for any modification of header, nonce, AAD, or ciphertext.
  6. Decoy-induced attacker uncertainty with zero risk to legitimate users.
  7. Non-destructiveness: no operation, ever, damages external state.

5. Mathematical Model of the Reactor

5.1 Attempt History

For a vault \(v\) over window \(\tau\), define attempt history

\[ A_v(\tau) = \{a_1, a_2, \dots, a_n\}, \]

where each attempt is the tuple

\[ a_i = (t_i,\ \text{ip}_i,\ \text{device}_i,\ \text{result}_i,\ \text{latency}_i,\ \text{state}_i,\ \text{integrity}_i). \]

Failed-attempt count:

\[ n_v(\tau) = \sum_{i=1}^{|A_v(\tau)|} \mathbf{1}[\text{result}_i = \text{fail}]. \]

5.2 Adaptive Computational Cost

\[ C(n) = C_0 \cdot \alpha^n, \qquad \alpha > 1. \]

Capped:

\[ \hat{C}(n) = \min\!\bigl(C_{\max},\ C_0 \cdot \alpha^n\bigr). \]

The cap \(C_{\max}\) guarantees legitimate users a bounded worst-case latency after accidental failures.

5.3 Adaptive Argon2id Memory

\[ M(n) = \min\!\bigl(M_{\max},\ M_0 \cdot 2^{n}\bigr). \]

Example schedule:

\(n\) \(M(n)\)
0 64 MiB
1 128 MiB
2 256 MiB
3 512 MiB
4 1024 MiB
5 2048 MiB

5.4 Adaptive Iterations and Parallelism

\[ I(n) = \min\!\bigl(I_{\max},\ I_0 \cdot \gamma^{n}\bigr), \qquad \gamma > 1. \]

Parallelism is held fixed:

\[ p(n) = p_0, \]

unless the implementation explicitly supports safe bounded scaling.

5.5 Defensive Delay

\[ D(n) = \min\!\bigl(D_{\max},\ D_0 \cdot \beta^n\bigr), \qquad \beta > 1. \]

\(D_{\max}\) is critical: it prevents the reactor itself from becoming a denial-of-service vector against legitimate users.

5.6 Decoy Branching and Uncertainty

\[ B(n) = \min\!\bigl(B_{\max},\ 2^{n}\bigr). \]

Attacker uncertainty in bits:

\[ U(n) = \log_2\!\bigl(B(n) + 1\bigr). \]

Probability that an attempt is routed to a decoy:

\[ P_{\text{decoy}}(n) = 1 - e^{-\kappa B(n)}, \]

where \(\kappa > 0\) controls decoy activation strength.

5.7 Reactor Risk Score

\[ R(n) = w_c \log\!\bigl(1+\hat{C}(n)\bigr) + w_m \log\!\bigl(1+M(n)\bigr) + w_i \log\!\bigl(1+I(n)\bigr) + w_d D(n) + w_b U(n) + w_a A(n) + w_h H_{\text{tamper}}. \]

All weights \(w_\bullet \geq 0\) are configurable per deployment.

5.8 Anomaly Score

\[ A(n) = \lambda_1 F_{\text{rate}} + \lambda_2 F_{\text{geo}} + \lambda_3 F_{\text{device}} + \lambda_4 F_{\text{timing}} + \lambda_5 F_{\text{integrity}} + \lambda_6 F_{\text{replay}}. \]

Components:

  • \(F_{\text{rate}}\): rate of repeated attempts.
  • \(F_{\text{geo}}\): abnormal location changes.
  • \(F_{\text{device}}\): unfamiliar device fingerprints.
  • \(F_{\text{timing}}\): scripted or unnatural timing distributions.
  • \(F_{\text{integrity}}\): vault-header or ciphertext tamper signals.
  • \(F_{\text{replay}}\): repeated stale blob submissions.

5.9 Decoy Entropy

\[ H_{\text{decoy}}(n) = -\sum_i p_i \log_2 p_i, \]

where \(\{p_i\}\) is the distribution over the \(B(n)\) decoy branches plus the real branch.

The reactor optimizes:

\[ \max\ U(n) \quad \text{subject to} \quad \text{false-positive harm to legitimate users} \leq \epsilon. \]

6. State Machine

6.1 States

State Meaning
OPEN Vault unlocked for the current authenticated session.
BACKOFF Linear/exponential cooldown after small numbers of failures.
QUARANTINE Stronger throttling; decoys may activate.
SCRAM_LOCK Emergency lock; session keys zeroized; recovery required to retry.
CONTAINMENT_LOCK Hardest internal containment; no further attempts processed until recovery.
RECOVERY_REQUIRED Awaiting out-of-band recovery credential or operator action.
ZEROIZE_SESSION Transient state during secure wiping of in-memory secrets.
DECOY_OPEN A wrong passphrase matched a decoy branch; vault returned a vault-local decoy payload.
TAMPER_EVIDENT HMAC or GCM verification failed; integrity violation recorded.

6.2 Transition Function

\[ \text{State}(n) = \begin{cases} \texttt{OPEN}, & R(n) < T_1 \\ \texttt{BACKOFF}, & T_1 \leq R(n) < T_2 \\ \texttt{QUARANTINE}, & T_2 \leq R(n) < T_3 \\ \texttt{SCRAM\_LOCK}, & T_3 \leq R(n) < T_4 \\ \texttt{CONTAINMENT\_LOCK}, & R(n) \geq T_4 \end{cases} \]

Tamper signals and decoy routing are overlaid on top of this base mapping:

  • If \(H_{\text{tamper}} = 1\), the state transitions to TAMPER_EVIDENT (and may escalate to SCRAM_LOCK).
  • If a decoy branch matches policy, DECOY_OPEN is entered for that single response.

6.3 Recovery Condition

\[ \text{RecoveryRequired} = \begin{cases} 1, & \text{State}(n) \in \{\texttt{SCRAM\_LOCK},\ \texttt{CONTAINMENT\_LOCK}\} \\ 0, & \text{otherwise.} \end{cases} \]

6.4 Zeroization Rule

If \(\text{State}(n) \in \{\texttt{SCRAM\_LOCK},\ \texttt{CONTAINMENT\_LOCK},\ \texttt{RECOVERY\_REQUIRED}\}\):

\[ \text{Zeroize}\bigl(K_{\text{session}},\ K_{\text{master}},\ K_{\text{enc}},\ K_{\text{mac}},\ K_{\text{decoy}},\ \text{plaintext\_cache},\ \text{passphrase\_buffer},\ \text{derived\_key\_buffer}\bigr). \]

Boundary. Zeroization affects only temporary internal secrets in process memory. It does not delete external files, user files, OS files, folders, other vaults, drives, or any network data.

6.5 State Diagram

              +--------+
              |  OPEN  |<-----------------------------+
              +---+----+                              |
                  | R(n) >= T1                        |
                  v                                   |
              +--------+      success / recovery      |
              |BACKOFF +----------------------------> +
              +---+----+                              |
                  | R(n) >= T2                        |
                  v                                   |
            +-----------+                             |
            |QUARANTINE +---------------------------> +
            +-----+-----+                             |
                  | R(n) >= T3                        |
                  v                                   |
           +-------------+   ZEROIZE_SESSION          |
           | SCRAM_LOCK  +-------> (wipe RAM secrets) |
           +------+------+                            |
                  | R(n) >= T4                        |
                  v                                   |
        +-------------------+      recovery           |
        | CONTAINMENT_LOCK  +-----> RECOVERY_REQUIRED-+
        +-------------------+

TAMPER_EVIDENT and DECOY_OPEN are overlay states triggered orthogonally to \(R(n)\) thresholds.

7. Cryptographic Pipeline (SxCrypt-Vault)

7.1 Key Derivation

Given passphrase \(P\), salt \(s\), current memory \(M(n)\), iterations \(I(n)\), parallelism \(p\):

\[ K_{\text{master}} = \text{Argon2id}\bigl(P,\ s,\ M(n),\ I(n),\ p\bigr). \]

7.2 Key Separation

\[ K_{\text{enc}} \,\|\, K_{\text{mac}} \,\|\, K_{\text{decoy}} \,\|\, K_{\text{audit}} \,\|\, K_{\text{policy}} = \text{HKDF\text{-}SHA512}\bigl(K_{\text{master}},\ \text{info} = \texttt{"SxCrypt-Vault-v2"}\bigr). \]

Distinct info strings can be used for each subkey in practice; the concatenation form is shown for compactness.

7.3 Authenticated Encryption

\[ C = \text{AES\text{-}256\text{-}GCM.Encrypt}\bigl(K_{\text{enc}},\ \text{nonce},\ \text{plaintext},\ \text{AAD}\bigr). \]

\[ \text{tag}_{\text{gcm}} = \text{GCM\_Tag}(C). \]

7.4 External Integrity

\[ \text{tag}_{\text{hmac}} = \text{HMAC\text{-}SHA512}\bigl(K_{\text{mac}},\ \text{header} \,\|\, \text{nonce} \,\|\, \text{AAD} \,\|\, C \,\|\, \text{tag}_{\text{gcm}}\bigr). \]

7.5 Tamper Verification

\[ \text{Verify}(\text{tag}_{\text{hmac}}) = \begin{cases} 1 & \Rightarrow \text{integrity valid} \\ 0 & \Rightarrow \text{enter } \texttt{TAMPER\_EVIDENT}\text{ or } \texttt{SCRAM\_LOCK}\text{ per policy.} \end{cases} \]

7.6 Decoy Derivation

For a wrong passphrase \(P_{\text{wrong}}\) that matches a decoy-routing rule:

\[ K_{\text{decoy},i} = \text{HKDF\text{-}SHA512}\Bigl(\text{Argon2id}\bigl(P_{\text{wrong}},\ s_i,\ M_{\text{decoy}},\ I_{\text{decoy}},\ p\bigr),\ \text{info} = \texttt{"SxCrypt-Decoy-i"}\Bigr). \]

Decoy rules. A decoy payload may be returned only if:

  1. Policy explicitly allows decoys.
  2. The decoy never executes code.
  3. The decoy never contacts a network.
  4. The decoy never modifies files.
  5. The decoy is clearly vault-local defensive deception.

8. Seal and Open Operations

8.1 Seal

Seal(plaintext, passphrase):
    vault_id   <- secure_random(128 bits)
    salt       <- secure_random(256 bits)
    nonce      <- secure_random(96 bits)             # unique per (key, message)
    header     <- {version, vault_id, salt, policy, n=0, kdf_params}
    K_master   <- Argon2id(passphrase, salt, M(0), I(0), p)
    (K_enc, K_mac, K_decoy, K_audit, K_policy)
               <- HKDF-SHA512(K_master, info="SxCrypt-Vault-v2")
    AAD        <- header
    C, tag_gcm <- AES-256-GCM.Encrypt(K_enc, nonce, plaintext, AAD)
    tag_hmac   <- HMAC-SHA512(K_mac, header || nonce || AAD || C || tag_gcm)
    blob       <- {header, nonce, C, tag_gcm, tag_hmac}
    atomic_write(blob)
    zeroize(passphrase, K_master, K_enc, K_mac, K_decoy, K_audit, plaintext)
    return vault_id

8.2 Open

Open(blob, passphrase):
    load header, nonce, C, tag_gcm, tag_hmac
    load reactor policy state (n, A(n), H_tamper, ...)
    compute C(n), M(n), I(n), D(n), B(n), U(n), A(n), R(n)
    if State(n) in {SCRAM_LOCK, CONTAINMENT_LOCK, RECOVERY_REQUIRED}:
        return reject("recovery required")
    sleep(D(n))
    K_master   <- Argon2id(passphrase, header.salt, M(n), I(n), p)
    (K_enc, K_mac, ...) <- HKDF-SHA512(K_master, ...)
    if not constant_time_eq(HMAC-SHA512(K_mac, header||nonce||AAD||C||tag_gcm), tag_hmac):
        H_tamper <- 1
        increment n; update state; zeroize; return tamper_evident
    try:
        plaintext <- AES-256-GCM.Decrypt(K_enc, nonce, C, tag_gcm, AAD=header)
        reset_counters_per_policy()
        state <- OPEN
        return plaintext
    except AuthFailure:
        increment n
        recompute R(n); update State(n)
        if policy.allow_decoy and decoy_match(passphrase):
            return open_decoy_branch()        # vault-local, inert
        zeroize(K_master, K_enc, K_mac, K_decoy, derived_buffers, passphrase)
        return reject(State(n))

All branches end in zeroization of temporary in-memory secrets.

9. SxCrypt-Core (Research Cipher)

This section describes a research-only AES-style SPN. It is included as a mathematical laboratory for studying SPN internals. Do not use SxCrypt-Core to protect real secrets.

9.1 State

\[ S_r \in \mathrm{GF}(2^8)^{4 \times 4}. \]

9.2 Round Function

\[ S_{r+1} = \text{AddRoundKey}\Bigl(\text{MixColumns}\bigl(\text{ShiftRows}\bigl(\text{SubBytes}(S_r)\bigr)\bigr),\ K_r\Bigr). \]

The final round may omit MixColumns, in the AES style.

Parameters: \(\{128, 192, 256\}\)-bit keys; \(\{12, 14, 16\}\) rounds.

9.3 S-Box

\[ S(x) = A \cdot \mathrm{Inv}_{\mathrm{GF}(2^8)}(x) \oplus c, \]

with:

  • \(A\) an invertible affine matrix over \(\mathrm{GF}(2)\).
  • \(c\) an affine constant byte.
  • Convention \(\mathrm{Inv}_{\mathrm{GF}(2^8)}(0) = 0\).

SxCrypt-Core default affine constant: \(c = \mathtt{0x53}\) (compare AES: \(c = \mathtt{0x63}\)).

Honest note. Changing the affine constant from \(\mathtt{0x63}\) to \(\mathtt{0x53}\) does not automatically make the cipher cryptographically stronger than AES. SxCrypt-Core’s purpose is study, not replacement.

9.4 Avalanche

For plaintext \(x\) and a one-bit perturbation \(x' = x \oplus e_i\), let \(E_K^{(r)}(\cdot)\) be the state after round \(r\). Define

\[ \Delta_r(i) = \mathrm{HammingDistance}\bigl(E_K^{(r)}(x),\ E_K^{(r)}(x')\bigr), \qquad \delta_r(i) = \frac{\Delta_r(i)}{\text{block\_size}}. \]

Ideal target:

\[ \mathbb{E}[\delta_r] \to \tfrac{1}{2}. \]

Strict Avalanche Criterion (SAC):

\[ \Pr\bigl[E_K(x)_j \neq E_K(x \oplus e_i)_j\bigr] \approx \tfrac{1}{2}. \]

Diffusion speed:

\[ \rho = \min\bigl\{r\ :\ |\mathbb{E}[\delta_r] - \tfrac{1}{2}| \leq \varepsilon\bigr\}. \]

Entropy per round:

\[ H(S_r) = -\sum_b p_b \log_2 p_b, \]

with \(p_b\) the empirical distribution of byte values in \(S_r\).

9.5 S-Box Metrics

Nonlinearity:

\[ \mathrm{NL}(S) = 2^{m-1} - \tfrac{1}{2} \max_{a, b \neq 0} \Bigl| \sum_x (-1)^{b \cdot S(x) \oplus a \cdot x} \Bigr|. \]

Differential uniformity:

\[ \mathrm{DU}(S) = \max_{a \neq 0,\, b}\ \#\{x\ :\ S(x) \oplus S(x \oplus a) = b\}. \]

Linear approximation bias:

\[ \mathrm{Bias}(a, b) = \Bigl| \Pr[a \cdot x = b \cdot S(x)] - \tfrac{1}{2} \Bigr|. \]

Algebraic degree:

\[ \deg(S) = \max_j \deg(S_j). \]

9.6 Diffusion Layer

Branch number:

\[ B_L = \min_{x \neq 0}\ \bigl(\mathrm{wt}(x) + \mathrm{wt}(L(x))\bigr). \]

Minimum active S-boxes over \(r\) rounds:

\[ A_r \geq \text{lower bound from the wide-trail strategy}. \]

A higher minimum active S-box count improves provable resistance to differential and linear cryptanalysis.

10. “More Than AES” — Stated Honestly

AES is a standardized block cipher. SxCrypt-Vault uses AES-256-GCM for the actual encryption layer. SxCrypt is more than AES at the system level because it adds:

  1. Adaptive Argon2id passphrase hardening.
  2. Memory-cost reaction \(M(n)\).
  3. Computational-cost reaction \(C(n)\).
  4. Defensive delay reaction \(D(n)\).
  5. Decoy branching \(B(n)\) and uncertainty \(U(n)\).
  6. Tamper-evident vault metadata.
  7. HMAC-SHA512 external integrity.
  8. HKDF-SHA512 key separation.
  9. SCRAM emergency lock.
  10. Containment lock.
  11. Session-key zeroization.
  12. Vault-local audit trail.
  13. Recovery-required policy.
  14. Explicit safety boundaries.
  15. Vault-local, non-destructive defense.

SxCrypt-Core is a research cipher used to study AES-style design. SxCrypt-Vault relies on AES-256-GCM for production protection. The innovation of SxCrypt is not replacing AES, but building a mathematically governed reactor-defense layer around encrypted vaults.

11. Safety Principles (Non-Negotiable)

SxCrypt is bound by the following invariants at all layers:

  1. Production encryption uses only standard, audited primitives.
  2. Custom cryptography is isolated to research mode.
  3. No self-propagation.
  4. No external file deletion.
  5. No destructive payloads.
  6. No hidden network activity.
  7. No filesystem traversal outside the vault path.
  8. No persistence mechanisms beyond the vault file itself.
  9. No privilege escalation.
  10. All containment is vault-local.
  11. All destructive-sounding metaphors (e.g. meltdown, SCRAM) map only to safe internal state changes — never to external damage.

12. Evaluation Methodology

12.1 Quantitative Metrics

  • Brute-force cost growth. Plot \(C(n)\), \(M(n)\), \(I(n)\), \(D(n)\), \(B(n)\), \(U(n)\), \(R(n)\).
  • State transitions. Plot \(R(n)\) against \(T_1, T_2, T_3, T_4\).
  • Legitimate-user usability.
    • Unlock latency at \(n=0\).
    • Unlock latency after accidental failures.
    • Recovery workflow time after SCRAM_LOCK.
  • Avalanche behavior. Round-by-round Hamming distance; convergence of \(\delta_r\) to \(1/2\).
  • Entropy. \(H(S_r)\) per round; byte distribution of round states.
  • S-box quality. Nonlinearity, differential uniformity, linear bias, algebraic degree; DDT and LAT heatmaps.
  • Diffusion layer. Branch number; minimum active S-box estimate; bit-dependency graph growth per round.
  • Tamper evidence. Bit-flip detection rate; header / ciphertext / nonce modification detection.
  • Decoy effectiveness. \(U(n)\), \(B(n)\), false-positive risk for real users.

12.2 Safety Validation Tests

  • Confirm no network calls are made by the vault binary.
  • Confirm no external file deletion.
  • Confirm no self-replication.
  • Confirm no file traversal outside the vault path.
  • Confirm zeroization affects only temporary internal secrets.

12.3 Required Plots

  1. \(C(n)\): computational cost reaction curve.
  2. \(M(n)\): memory reaction curve.
  3. \(I(n)\): iteration reaction curve.
  4. \(D(n)\): delay reaction curve.
  5. \(B(n)\): decoy branching curve.
  6. \(U(n)\): attacker uncertainty curve.
  7. \(R(n)\): reactor risk score with thresholds \(T_1, T_2, T_3, T_4\).
  8. State transition diagram.
  9. Failed attempts vs. containment level.
  10. Legitimate user latency vs. attacker cost.
  11. Round avalanche curve \(\delta_r\).
  12. Entropy per round \(H(S_r)\).
  13. S-box differential distribution heatmap (DDT).
  14. S-box linear approximation heatmap (LAT).
  15. Key schedule visualization.
  16. Reactor heatmap over time.
  17. Tamper-detection success chart.

13. Implementation Requirements

  1. Constant-time comparison for all MAC and tag checks.
  2. Cryptographically secure randomness for salts, nonces, vault IDs, decoy IDs.
  3. Never reuse an AES-GCM nonce with the same key.
  4. Version every vault header.
  5. Bind header metadata into AAD for AES-GCM.
  6. Separate keys via HKDF; never use a single derived key for multiple purposes.
  7. Zeroize sensitive memory wherever the language and runtime permit.
  8. Safe file permissions on vault files (owner-only by default).
  9. Atomic writes for vault updates (write-temp + rename).
  10. Vault-local, privacy-preserving audit logs.
  11. Dangerous operations impossible by design — not merely disabled by configuration.
  12. Separate research mode and production mode. Production mode must use audited primitives only.

14. Operational Boundaries

SxCrypt’s adaptive defenses must always satisfy:

  • Maximum delay \(D_{\max}\) is bounded so accidental failures do not produce a denial of service against the legitimate user.
  • Memory cap \(M_{\max}\) is bounded by the target deployment hardware to prevent self-induced exhaustion.
  • Cost cap \(C_{\max}\) is bounded so recovery workflows remain feasible.
  • Recovery is always possible via a documented, out-of-band recovery credential.
  • No state transition, including CONTAINMENT_LOCK, ever performs an action outside the vault container.
